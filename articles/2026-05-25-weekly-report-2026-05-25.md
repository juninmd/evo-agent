## Deep Dive Técnico: Navegação Instantânea no GitHub Issues

O post mais técnico da semana veio do time de GitHub Issues, que documentou em detalhes como modernizou a navegação da aplicação — um caso de estudo em **performance percebida**.

### O Problema

Navegar entre issues, PRs e projetos no GitHub envolve **múltiplas requisições de rede**, renderização de componentes pesados e reidratação de estado. O time identificou que a **latência percebida** era o maior ponto de dor dos usuários, com tempos de navegação muito acima do ideal.

### A Arquitetura: Stale-While-Revalidate + Hot Cache + Prefetching

A solução combina três técnicas em camadas:

```
[IntersectionObserver] → prefetch viewport-intent
         ↓
    [In-Memory Hot Cache] → resposta instantânea (P10: 70ms)
         ↓         ↻ (stale)
    [IndexedDB] → backfill persistente
         ↓
    [Service Worker] → SWR fetch + annotated-header signaling
         ↓
    [React.lazy + Route Splitting] → carregamento sob demanda
```

**1. Stale-While-Revalidate com IndexedDB**

O padrão SWR permite servir uma resposta **instantânea** do cache (mesmo que "stale") enquanto uma requisição de rede revalida o dado em background. O GitHub Issues usa IndexedDB como cache persistente:

- **Hit no hot cache (in-memory):** resposta em ~70ms (P10)
- **Hit no IndexedDB (backfill):** resposta em ~120ms (P25) com revalidação em background
- **Miss total (rede):** resposta em ~700ms (P50)
- **Cache-hit ratio target: 96%+** com preheating

**2. Prefetching com IntersectionObserver**

O time implementou **viewport-intent prefetching**: quando o mouse ou o foco se aproxima de um link (medido por IntersectionObserver), o Service Worker inicia o fetch preventivamente. Isso significa que quando o usuário **clicar**, o dado já está no cache.

**3. Service Worker com Annotated-Header Signaling**

O Service Worker enxerga headers customizados que sinalizam se uma navegação deve ser tratada como **"instantânea"** (cache-first) ou **"nova"** (hard navigation). Isso evita o problema clássico de servir páginas obsoletas em navegações críticas.

**4. React.lazy + Route Splitting**

Cada rota é um chunk separado, carregado sob demanda. Combinado com o prefetching, o usuário tem a **ilusão de que toda a aplicação está pré-carregada**, sem o custo de banda de um bundle monolítico.

O padrão é replicável em qualquer app React com roteamento:

1. **Meça antes de otimizar** — o time do GitHub usou RUM (Real User Monitoring) para identificar os gargalos reais.

### 2. Stale-While-Revalidate Cache com Hot Layer

```typescript
class HotCache<T> {
  private memory = new Map<string, { data: T; expires: number }>();
  private db: IDBPDatabase<Schema>;

  async get(key: string): Promise<{ data: T; stale: boolean }> {
    const hot = this.memory.get(key);
    if (hot && hot.expires > Date.now()) return { data: hot.data, stale: false };
    const cold = await this.db.get("cache", key);
    if (cold) {
      this.memory.set(key, { data: cold.data, expires: Date.now() + 30_000 });
      return { data: cold.data, stale: true };
    }
    return { data: await this.fetch(key), stale: false };
  }
}
```

### 3. Multi-Agent MCP Orchestrator com Phase Gates

```typescript
interface Phase<TInput, TOutput> {
  name: string;
  execute(input: TInput): Promise<TOutput>;
  gate(output: TOutput): Promise<boolean>; // rollback se false
  rollback(output: TOutput): Promise<void>;
}

class Orchestrator {
  async run<T>(phases: Phase<any, any>[], input: T): Promise<Result> {
    const completed: Phase<any, any>[] = [];
    for (const phase of phases) {
      try {
        const output = await phase.execute(input);
        if (!(await phase.gate(output))) {
          await this.rollbackAll(completed);
          return { status: "failed", phase: phase.name };
        }
        completed.push(phase);
      } catch (e) {
        await this.rollbackAll(completed);
        return { status: "error", error: e };
      }
    }
    return { status: "success" };
  }
}
```
